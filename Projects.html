Self Balancing Robot

Two‑wheeled self‑balancing robot. IMU feedback and PID control to stay upright and respond to Bluetooth commands.

Important Skills Used

  • PID Control System
  • Complimentary Filtering
  • Real Time Angle Calculation
  • I2C Communication Protocol
  • Embedded C++
Project Description (Click to Expand)

This project involved designing and implementing a two‑wheeled self‑balancing robot built around the Arduino Nano 33 BLE, programmed in C++ using the Arduino framework. The robot’s primary sensing subsystem was based on the BMI270 IMU, which provides both accelerometer and gyroscope measurements. These two data sources were fused using a complementary filter, allowing the system to combine the short‑term stability of the gyroscope with the long‑term accuracy of the accelerometer. The resulting filtered pitch angle served as the input to a high‑frequency PID control loop, which continuously adjusted motor output to maintain balance. The PID controller computed a corrective torque based on the angle error, and this output was mapped to motor PWM duty cycles, enabling smooth and responsive stabilization. 

The actuation subsystem consisted of two Pololu 4741 DC gear motors driven by DRV8833 dual H‑bridge motor drivers. Each motor was controlled independently using PWM signals generated by the Arduino, with the drivers configured in soft‑decay mode to improve stability during rapid direction changes. This configuration allowed the robot to apply fine‑grained torque adjustments, which is essential for balancing and for executing controlled forward, backward, and turning motions. The motor control logic integrated the PID output with additional offsets for directional commands, ensuring that user‑initiated movements did not interfere with the balancing loop. 

For wheel rotation sensing, the robot incorporated AS5600 magnetic encoders, each paired with a diametric permanent magnet mounted on the motor shaft. The AS5600 is a Hall‑effect–based rotary encoder that measures the angular position of the rotating magnetic field. Because both encoders share the same fixed I²C address, a TCA9548A I²C multiplexer was used to route communication through the Arduino’s single hardware I²C bus. This approach preserved the high communication speed required to maintain a short control‑loop cycle time, which is critical for stable balancing. Although encoder feedback was not used in the final PID loop, the sensors were valuable for tracking wheel rotation during commanded movements, enabling the robot to estimate how far it had turned or traveled when executing user‑initiated actions. 

Together, these subsystems formed a tightly integrated embedded control platform capable of real‑time sensing, computation, actuation, and communication. The combination of IMU‑based angle estimation, high‑speed PID control, PWM motor actuation, and optional encoder‑based motion tracking allowed the robot to balance autonomously while responding to wireless directional commands with precision.

This was a group project completed with two of my peers, I was responsible for all embedded hardware, firmware, and control system development.

Magnetic Field Controlled Robot

Wireless inductor‑controlled robot capable of user‑driven motion and maintaining a fixed distance to the transmitter

Receiver Hardware Diagram

Important Skills Used

  • Custom Wireless Communication Protocol
  • Precision ADC Integration
  • H-Bridge Motor Driver Control
  • Embedded C++
  • Embedded State Machine Design
Project Description (Click to Expand)

The transmitter subsystem was built around an EFM8 microcontroller, which generated a 16 kHz excitation waveform to drive a resonant LC tank circuit through an H‑bridge. The microcontroller produced the alternating signal using a timer‑driven pin‑toggle routine, ensuring a stable magnetic field at the resonant frequency of the inductor–capacitor network. User input was provided through a Wii Nunchuck connected via I²C, enabling joystick‑based directional commands and mode selection. Commands were encoded by briefly interrupting the magnetic field for specific time intervals, and an LCD display provided feedback on the active mode and transmitted instruction. 

The receiver subsystem was built around an STM32 microcontroller and used two matched RLC sensing circuits to detect the transmitted magnetic field. Each inductor fed into a peak‑detector and op‑amp stage, producing DC voltages proportional to the magnetic field strength at each coil. These voltages were sampled using the STM32’s ADC and used for both command decoding and directional tracking. In beacon mode, the robot determined its orientation by comparing the relative amplitudes of the left and right peak‑detector outputs: the coil with the higher amplitude was physically closer to the transmitter’s magnetic field, and the difference between the two voltages indicated both direction and required correction. This amplitude‑based comparison allowed the robot to align itself with the transmitter and maintain a consistent distance. The transmitter firmware implemented a state‑machine architecture that continuously sampled the Nunchuck inputs, determined the requested movement or mode, and generated the appropriate magnetic‑field interruption pattern. Timer2 produced the base oscillation, while timed pauses were created using millisecond‑scale delay routines to encode forward, backward, left, right, and mode‑switch commands. This ensured consistent, repeatable timing signatures that the receiver could reliably interpret. 

The receiver firmware measured the duration of magnetic‑field interruptions using the SysTick timer and mapped each timing pattern to a specific movement instruction. In beacon mode, the firmware used the difference in peak‑detector amplitudes to determine whether the robot should turn left, turn right, move forward, or move backward. Large amplitude differences triggered coarse steering adjustments, while small differences triggered fine‑tuning movements at reduced motor speed. Additional logic supported soft‑turning, a “return home” function that replayed stored commands in reverse, and filtering to stabilize behavior in noisy magnetic environments. Together, the transmitter and receiver formed a complete low‑frequency magnetic communication system capable of reliable short‑range control and autonomous tracking.

This was a group project completed with three of my peers. I was responsible for receiver embedded firmware and hardware, and the wireless communication protocol that was used.

Reflow Oven Controller

Toaster oven controlled by EFM8 ​microcontroller, UI for creating and selecting soldering profiles.

Project Description (Click to Expand)

This project involved designing and implementing a complete reflow oven controller using an EFM8 microcontroller, programmed entirely in assembly language. The controller executed a full reflow soldering profile by integrating temperature sensing, PWM‑based power regulation, user interaction, safety enforcement, and audio feedback into a cohesive embedded system. 

Temperature measurement was performed using a K‑type thermocouple connected to a precision OP07 operational amplifier configured as a differential gain stage. Because thermocouple voltages are extremely small, the circuit included a carefully selected gain and offset adjustment to ensure accurate readings across the full reflow temperature range. An LM335 temperature sensor provided ambient temperature compensation, and both signals were digitized using an MCP3008 SPI‑based 10‑bit ADC. The microcontroller sampled these values and computed the oven temperature in real time, forming the basis of the thermal control loop. 

Heating power was regulated using pulse‑width modulation (PWM) generated inside the Timer2 interrupt service routine. The PWM operated with a one‑second period and ten discrete duty‑cycle levels, driving a solid‑state relay that switched the oven’s heating element. A finite‑state machine (FSM) implemented the standard reflow stages—preheat, soak, reflow, and cooling—advancing based on temperature thresholds and timing constraints defined by the selected profile. Each state dynamically adjusted the PWM duty cycle to track the desired temperature curve while maintaining safe operating conditions.

The controller supported four configurable and saveable reflow profiles, stored in on‑chip flash memory. A structured menu system allowed users to select, edit, and validate profiles using onboard pushbuttons and a character display. Safety checks were integrated throughout the interface and control logic, preventing invalid profiles from running and automatically aborting the process if unsafe conditions were detected, such as insufficient heating rate or excessive temperature rise. 

An independent audio subsystem provided spoken feedback during operation. This system used a 25Q32 SPI flash memory to store WAV‑encoded audio clips, a MCP1700 voltage regulator, an LM386 audio amplifier, and a speaker. A dedicated sound FSM sequenced the playback of temperature announcements and state transitions without blocking the main control loop, ensuring continuous and responsive system behavior. 

Together, these components formed a robust embedded system capable of accurately measuring temperature, regulating heating power, enforcing safety constraints, and providing clear user interaction. The combination of analog signal conditioning, SPI‑based data acquisition, PWM power control, non‑volatile configuration storage, and real‑time state‑machine logic enabled the controller to execute reliable and repeatable reflow cycles suitable for PCB assembly. 

This was a group project completed with three of my peers. I was responsible for all embedded firmware that was developed.
